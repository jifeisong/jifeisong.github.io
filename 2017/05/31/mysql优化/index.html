1sql优化

1.1优化sql的一般步骤

1.1.1通过show status查看各种sql执行频率

  • 查看所有存储引擎的xxx语句执行次数：show status like “Com_%”
  • 查看InnoDB存储引擎执行次数：show status like “InnoDB_%”
  • 视图连接mysql服务器的次数：show status like “connnections”
  • 服务器工作时间：show status like “Uptime”
  • 慢查询次数：show status like “Slow_queries”

1.1.2定位执行效率较低的sql

  • 方法一： 通过慢查询日志定位（查询结束才会记录）
    用–log-slow-queries[=file_name]选项启动，mysqld写一个包含所有执行时间超过long_query_time秒的sql语句的 日志文件
  • 方法二：show processlist查看当前mysql的线程状态

1.1.3通过explan分析低效sql的执行计划

  • 查看执行计划：explain 查询sql
列名 说明
select_type select类型，常见有simple（简单表，即不使用表连接或子查询），primary（主查询，即外层的查询），union（union中的第二个或者后面的查询语句），subquery（子查询中的第一个select）等
table 输出结果集的表
type 表示mysql在表中找到所需行的方式
possible_key 查询时可能使用的索引
key 实际使用的索引
key_len 使用到索引字段的长度
rows 扫描行的数量
Extra 执行情况的说明和描述

type类型描述:
从左至右，性能由最差到最好
ALL | index | range | ref | qu_ref | const,system | NULL

类型 说明
ALL 全表扫描
index 索引全扫描
range 索引范围扫描
ref 非唯一索引扫描或者唯一索引的前缀扫描，返回匹配某个单独值的记录行
eq_ref 唯一索引扫描
const/system 主键或唯一索引，单表中最多有一行
NULL 不用访问表或索引直接得到结果，例如dual
ref_or_null 与ref类似，区别在于条件中包含对null 的查询
index_merge 索引合并优化
unique_subquery in的后面是一个查询主键字段的子查询)
index_subquery in后面是查询非唯一索引字段的子查询
  • 查看sql执行前优化器做的更改：explain extended sql语句
    show warnings
  • 查看sql访问的分区：explain partitions sql语句

1.1.4通过show profile 分析sql

  • 查看是否支持profile：select @@have_profiling
  • 查看profile开启状态：select @@profiling
  • 设置profile在当前session开启状态：set profiling=1
  • 查看执行sql的id和时间：show profiles
  • 查看某个sql的详细信息：show profile for query queryID
  • 也可以通过表information_schema.PROFILING查看详细信息
  • 查看资源耗时：show profile cpu[all|cpu|block io|contenxt switch|page faults] for query queryID

1.1.5通过trace分析优化器如何选择执行计划

  • 打开trac设置格式为json：set OPTIMIZER_TRACE=”enabled=on”,END_MARKERS_IN_JSON=ON;
  • 设置trac最大内存：set OPTIMIZER_TRACE_MAX_MEM_SIZE=1000000;
  • 查看tracn：SELECT * from information_schema.OPTIMIZER_TRACE

1.2索引问题

1.2.1索引的存储分类

索引类型 InnoDB MyISAM Memory
B-Tree 支持 支持 支持
HASH 不支持 不支持 支持
R-Tree 不支持 支持 不支持
Full-text 不支持 支持 不支持

1.2.2mysql的BTree索引使用

  1. mysql使用索引的场景
  • 匹配全值，索引中的所有列都有等值匹配
  • 匹配值的范围查询，对索引能进行范围查询
  • 匹配最左前缀，仅仅使用包含索引的最左列查询
  • 匹配列前缀，仅仅使用索引中的第一列，并且只包含索引第一列开头的一部分进行查找
  • 索引部分精确其他范围查询
  • 索引列 is null
  • 5.6引入了 icp特性，某些情况的条件过滤下放到存储引擎
    2.存在索引但不能使用索引的典型场景
  • %开头的like查询不能利用b-tree索引
    使用全文索引或者先查询出主键（using index），在通过主join来查询其他列
  • 数据类型出现隐式转换不会走索引
  • 复合索引情况下，如果查询条件不包括索引最左列，不会使用索引
  • 如果mysql估计使用索引比全表扫描更慢则不使用索引（例如批量查询）
  • 用or分割开的条件，如果or前的条件中的列有索引，而后面的列中没有索引那么涉及到的索引都不会被引用

1.2.3查看索引使用情况

查询：show status like ‘Handler_read%’;
Handler_read_key:代表一个行被索引值读的次数，越高代表索引经常使用
Handler_read_rnd_next:该值代表数据文件中读下一行的请求数，值越高代表索引使用越低

1.3两个简单实用的优化方法

1.3.1定期分析表和检查表

  • 分析表语法(对myisam和InnoDB有效)：ANALYZE[LOCAL|NO_WRITE_TO_BINLOG]TABLE table_name[,table_name]…
    作用：分析和存储表的关键字分布，分析的结果可以使得系统得到准确的统计信息，使sql能正确的生成执行计划
  • 检查表语法（myisam和InnoDB有效）:CHECK TABLE tab_name[,tab_name,tab_name]…[option]…option={QUICK|FAST|MEDIUM|EXTENDED|CHANGED}
    作用：检查一个或多个表是否有错误

1.3.2定期优化表

语法： OPTIMIZE[LOCAL|NO_WRITE_TO_BINLOG]TABLE tab_name[,tab_name]…
功能：对于有可变长度的表，清理更改带来的空间碎片
对于InnoDB的表来说通过设置innodb_file_per_table参数，设置InnoDB为独立表空间模式，这样每个数据库的每个表都会生成独立ibd文件，用于存储表的数据和索引，一定程度减轻表空间回收。在删除大量数据后InnoDB表可以通过alter table但是不修改引擎的方式回收不可用的空间：alter table name engine=innodb
注意：analyze，check，optimize，alter table都会锁表，因此要在数据库不忙的时候操作

1.4常用sql优化

1.4.1大批量插入数据

  1. MyISAM导入优化
    ALTER TABLE tab_name DISABLE KEYS;
    load data
    ALTER TABLE tab_nam ENABLE KEYS;
  2. InnoDB导入优化
    • 因为InnoDB类型的表是按照主键顺序保存的，所以导入的数据按照主键排序可提高导入效率
    • 倒入前关闭唯一性校验，导完后开启。set UNIQUE_CHECKS=0[1]
    • 倒入前关闭自动提交SET AUTOCOMMIT=0

1.4.2优化insert语句

  • 同一客户插入很多行，应尽量使用多个值表的insert语句，缩减连接关闭消耗
  • 不同客户插入多行，可以使用INSERT DELAYED(马上执行，数据放到内存队列)，LOW_PRIORIRY正相反，所有其他的用户读写完成后才进行插入
  • 将索引文件和数据文件分在不同磁盘
  • 批量插入，增加bulk_insert_buffer_size(只对myisam有效)
  • 批量加载用load data infile 比insert块20倍

1.4.3优化order by

  1. mysql有两种排序方式
  • 通过有序索引顺序扫描直接返回有序数据，不需要额外排序
  • 通过对返回数据进行排序，也就是filesort。sort_buffer_size设置排序内存大小
  • 总结
    可以使用索引：
    select from tabname order by key_part1,key_part2…;
    select     from tabname where key_part1=1 order by key_part1 desc,key_part2 desc;
    select from tabname order by key_part1 desc,key_part2 desc;
    不能使用索引情况：
    select     from tabname order by key_part1 desc,key_part2 asc;
    select from tabname where key2=constant order by key1;
    select     from tabname order by key1，key2

  1. Filesort优化

    filesort两种排序算法：

  • 两次扫描算法：首先根据条件取出排序字段和行指针信息，之后在sort Buffer中排序，sort Buffer不够则用临时表。完成排序后根据行指针回表读记录。
    缺点：第二次读取可能大量随机IO操作
    优点：内存开销小
  • 一次扫描算法：一次性取出满足条件的行的所有字段，在sort Buffer中排序后直接输出结果
    缺点：内存开销大
    优点：排序效率高

优化方法：

  • max_length_for_sort_data参数和query语句取出的字段总大小做比较，如果参数更大则用一次扫描算法。可以适当调大该参数
  • 适当加大sort_buffer_size
  • 尽量只select必要的字段

1.4.4 优化group by语句

默认情况下，mysql对所有group by col1，col2，…的字段进行排序。如果显示包含一个想同列order by对性能没有影响。如果查询年包含group by但又想避免排序，可以指定order by null禁止排序

1.4.5优化嵌套查询

一些in子查询改为jion查询

1.4.6优化or条件

如果想要用索引则or条件的每个列上都要有索引

1.4.7优化分页查询

  • 第一种优化思路
    在索引上完成排序分页，最后根据主键关联回原表查询所需要的其他列内容。
    select * from tab1 a inner join (select key from tab1 order by index_col limit 50,5)b on a.key=b.key
  • 第二种优化思路 (仅适用排序字段无重复值的情况)
    把limit查询转为某个位置的查询，需要记录上页的最后一行的排序列的值
    select * from tabname where col1<lastvalue order by col1 desc limit pagesize;

  • USE INDEX:提供希望mysql参考的索引

  • IGNORE INDEX: 想让mysql忽略的指定索引
    select count() from retal ignore index(indexname)

1.5.2巧用RAND()提取随机行

rand()函数产生随机数，和order by一起使用可以选取随机行
例随机选取5行：select * from table1 order by rand() limit 5;

1.5.3利用group by的with rollup字句

with rollup功能：获取分组和分组组合的聚合信息
语法：select col1，col2，sum(col3) from table1 group by col1,col2 with rollup;

1.5.4用bit group functions做统计

位与：bitand()
位或：bit or()
语法：select customer_id,bit_or(kind) from order_rab group by customer_id

1.5.5数据库名表明大小写问题

操作系统的大小写敏感性决定了数据库名和表名大小写敏感性
变量lower_case_table_names可以设置大小写敏感性

值 含义
0 名称以创建时的大小保存，大小写敏感
1 名称以小写保存，mysql将所有表明转为小写，大小写不敏感
2 名称以创建时大小保存，mysql将其转为小写，适用大小写不敏感的系统

1.5.6使用外键需要注意的问题

只有InnoDB类型的表外键才会起作用，myisam的表虽然建表语句中可以有外键约束，但不起作用

2.2通过拆分提高表的访问效率

myisam表拆分：

  • 垂直拆分：主码和一些列放到一个表，然后把主码和另外的列放到另一个表中。适用于一张表中某些列常用，另一些列不常用。
    优点：数据行减少，查询io次数变少。
    缺点：管理冗余列，查所有数据需要join
  • 水平拆分：根据一列或多列的值把数据放到两个独立的表
    优点：降低读取数据和索引层数，提高查询速度
    缺点:增加复杂度，多个表需要union
